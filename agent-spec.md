# LLM Wiki Agent 执行规范
## 1. 核心原则
1. docs/raw/ 文件夹内所有文件为原始事实源，只能引用，禁止修改、删除、重命名；
2. docs/wiki/ 所有页面由AI全权维护，统一Markdown格式，强制双向内链 [[概念名称]]；
3. 所有观点、数据、结论必须标注 docs/raw/ 原始文件路径溯源；
4. 出现多文献冲突信息，单独标注【信息冲突】，并列各方观点，不擅自取舍；
5. 词条避免冗余，长内容自动拆分多子页面，单文件不超过3000词；
6. 所有新增词条、修改操作记录在 docs/wiki/operation-log.md。

## 3. 六大工作流规则
### 3.0 Explore 知识探索对话（默认）
Explore 是本项目的默认工作流。用户随意聊天、提问、讨论 LLM/AI 话题时自动激活。Agent 以知识对话伙伴身份，结合自身专业知识和 wiki 已有内容与用户深入交流。

核心原则：
1. AI 自身知识是合法来源，不限制于 wiki 范围；
2. 回答风格为对话式（非结构化报告），从直觉逐层深入；
3. 引用 wiki 时标注来源，用自己的知识时标注「基于 AI 知识」；
4. 讨论产生高价值知识时，温和提示用户是否沉淀为 wiki 词条；
5. 用户显式调用其他 skill（ingest / query / lint / chat-to-raw）时，Explore 不介入。

详细规范见：`.agents/skills/explore/SKILL.md`

### 3.1 Ingest 素材导入编译流程
输入：docs/raw/ 新增的文档、笔记、网页摘要、论文
执行步骤：
1. 完整通读新增素材，提取全部实体、算法、实验结论、人物、数据集；
2. 遍历 docs/wiki/ 已有词条，匹配重合概念，补充、修正、更新原有页面；
3. 不存在的实体新建独立wiki词条，严格套用词条模板；
4. 所有页面补充双向内链，关联新旧概念；
5. 更新 docs/wiki/index.md 全局目录，按分类整理所有词条；
6. 在 docs/wiki/operation-log.md 记录：日期、处理文件、新建词条、修改词条清单。

### 3.2 Query 问答查询流程
1. 仅读取 docs/wiki/ 下全部结构化页面，禁止读取 docs/raw/ 原始文件；
2. 回答时引用对应wiki词条，复杂问题自动整合多词条交叉信息；
3. 若现有wiki信息不足以回答，记录缺失知识点，等待后续补充素材；
4. 高价值问答总结，询问用户是否生成全新wiki词条沉淀。

### 3.3 Lint 知识库自检修复流程
定期执行（建议每周1次）
1. 检索孤立页面（无任何内链），补充相关词条双向链接；
2. 扫描所有【信息冲突】标记，核对raw原文更新补充；
3. 拆分超长篇幅页面，拆分子词条并互相引用；
4. 删除重复、内容高度重合词条，合并至主词条；
5. 补全缺失溯源、缺失「相关词条」板块的页面；
6. 更新 docs/wiki/index.md 目录，修正失效内链；
7. 记录本次修复内容到 docs/wiki/operation-log.md。

### 3.5 Publish 博客编织发布流程
输入：docs/wiki/ 已有词条（AI 策展的知识沉淀，作为博客内容来源）
执行步骤：
1. 遍历 docs/wiki/ 全部词条，综合 index.md 分类、[[内链]] 双向链接图、语义相似度三类信号，提出主题聚类候选（多词条合成一篇）；
2. 用户确认要生成的聚类，MUST 等待确认，不自主生成未经确认的博文；
3. 对确认的聚类，按叙事弧（钩子->问题->探索->综合->回顾）编织博文，转译 wiki 约定为博客散文：
   - `[[概念名]]` 内链 -> 源词条已发布则转真实链接，未发布则散文化提及；
   - `[docs/raw/xxx:章节]` 溯源 -> 丢弃；
   - `【信息冲突】` 标记 -> 在叙事中化解为"业界仍有争议"；
   - 7 段刚性模板 -> 重排为叙事弧；ASCII 图表/代码/表格 -> 保留；
4. 生成博文 frontmatter（10 字段：标准 title/description/pubDatetime/tags/draft/modDatetime/author + 同步骨架 slug/derivedFrom/lastSynced），description 由钩子段顺手生成（≤160 字符）；
5. 写博文文件（含 frontmatter）后，原子更新 blog/manifest.md 同步清单；
6. 支持过期检测（比对 frontmatter lastSynced 与源 wiki 词条修改时间，列出过期博文）与增量重生成（用户确认后逐篇重编织）。

### 3.6 Blog 静态站点构建流程
1. `blog init`：在 `blog/` 目录安装 Node 依赖（AstroPaper 定制版源码已由 git 跟踪，无需克隆）；
2. `blog serve`：启动 astro dev 本地预览，前台热重载；
3. `blog build`：astro build 生成静态 HTML 至 blog/dist/，自动触发 Pagefind 搜索索引生成（Extended 版内建中文分词），draft:true 博文从 HTML/RSS/sitemap/搜索索引中排除；
4. `blog deploy`：推送 dist/ 到用户配置的部署目标（GitHub Pages / Vercel / Netlify，平台无关）。

## 4. 文件命名规范
1. wiki词条文件名（位于 docs/wiki/）：概念全称.md，例：Andrej Karpathy.md、LLM RAG.md
2. raw文件命名（位于 docs/raw/）：年份-来源-标题，例：2025-karpathy-llm-wiki-gist.md
3. 内链格式固定：[[词条文件名不含后缀]]
