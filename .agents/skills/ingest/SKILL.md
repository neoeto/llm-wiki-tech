# Ingest（素材导入编译）

**严格遵循项目根目录 `agent-spec.md` 全部规则。**

将 `docs/raw/` 下的原始素材编译为 `docs/wiki/` 下的结构化词条。这是将非结构化知识转化为可检索、可链接、可溯源 wiki 知识体的入口。

## 先决条件

**CRITICAL — 开始前 MUST 先完整读取以下文件：**

1. **`agent-spec.md`** — 核心原则、标准词条模板、三大工作流规则、文件命名规范
2. **`docs/wiki/index.md`** — 全局目录，了解当前分类框架
3. **`docs/wiki/operation-log.md`** — 操作历史，了解已有记录格式
4. **`docs/wiki/` 下所有现有词条页面** — 遍历全部 `.md` 文件，建立内容映射
5. **待处理 raw 素材** — 完整读取目标文件

> 反例自检：未读完现有 wiki 全部词条就开始提取 → 必然导致重复创建、漏更新。**遍历在先，提取在后。**

## 核心流程

```
docs/raw/新素材到达
      │
      ▼
┌─────────────────────────────────────────┐
│ Phase 1: 完整阅读 + 实体提取              │
│ - 提取全部：实体、技术概念、人物、架构、    │
│   方法论、论文、数据集、实验结论            │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ Phase 2: 对比 docs/wiki/ 已有词条        │
│ - 匹配重合概念 → 标记更新/补充            │
│ - 未覆盖概念 → 标记新建                   │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ Phase 3: 执行变更                        │
│ - 更新已有词条（补充、修正、扩展）         │
│ - 新建词条（严格套用标准模板）            │
│ - 所有页面补全双向内链 [[概念名]]         │
│ - 每条结论标注 raw 溯源                  │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ Phase 4: 全局一致性                      │
│ - 更新 docs/wiki/index.md 分类目录        │
│ - 追加 docs/wiki/operation-log.md 操作记录 │
│ - 输出变更汇总清单                       │
└─────────────────────────────────────────┘
```

## Phase 1 — 素材阅读与实体提取

### 操作规范

1. **完整通读** raw 文件，不跳读、不预判
2. **提取全部实体类型**：

| 类型 | 提取内容 |
|------|---------|
| 技术概念 | 术语、缩写、方法论、范式 |
| 人物 | 研究者、作者、贡献者 |
| 架构 | 系统架构、框架、分层结构 |
| 方法论 | 训练方法、优化策略、评估方法 |
| 论文 | 论文标题、作者、核心贡献 |
| 数据集 | 数据集名称、规模、用途 |
| 实验结论 | 量化结果、对比结论、消融实验 |
| 时间线 | 发布年份、版本里程碑 |

3. **摘录关键原文段落**，标注所在行/章节，保留原始上下文
4. **建立实体间关系图**：技术A依赖技术B、人物C提出方法D

### 输出产物
- 实体提取清单（概念名 + 类型 + raw 源位置）
- 关系映射表（A → B 依赖/对比/衍生关系）

## Phase 2 — 差异化对比

### 操作规范

1. **逐条对比**已提取实体与现有 wiki 词条标题、内容
2. **三种匹配结果**：

| 匹配结果 | 处理方式 |
|---------|---------|
| **完全命中** | 已有词条 → 检查是否需要补充新信息、修正旧信息 |
| **部分重叠** | 已有词条相关内容 → 扩展补充，添加新维度 |
| **未命中** | wiki 不存在该词条 → 标记「待新建」 |

3. **信息冲突检测**：
   - 若 raw 素材与现有 wiki 对同一事实的描述矛盾
   - **不擅自取舍**，在相关词条中标注 `【信息冲突】`，并列双方观点
   - 格式：`【信息冲突】docs/raw/xxx.md 第X章提出……；当前词条此前记载……；待进一步核实。`

### 输出产物
- 更新清单（现有词条 → 需补充章节）
- 新建清单（待创建词条名）
- 冲突标记清单（冲突位置 + 双方观点）

## Phase 3 — 执行变更

### 3.1 更新已有词条

对 Phase 2 标记「更新」的词条：

1. **读取完整原文**，不可仅读缓存
2. **在对应章节追加**新信息，保留原有内容结构
3. **不改变**原有章节顺序和核心定义
4. 所有新增观点末尾追加 `[docs/raw/xxx.md:章节名]` 溯源
5. 检查并补全双向内链

### 3.2 新建词条

对 Phase 2 标记「新建」的实体，严格套用以下模板（出自 `agent-spec.md`）：

```markdown
# {{词条名称}}

## 1. 基础定义
一句话精准定义该概念、模型、技术、人物、论文。

## 2. 核心原理 / 核心内容
分层拆解核心逻辑、架构、流程、公式。

## 3. 关键特性 & 优缺点
### 优势
- 

### 局限
- 

## 4. 适用场景
列出落地、学习、工程使用场景。

## 5. 对比关联
与相似概念、竞品模型做横向对比，添加双向内链 [[XXX]]。

## 6. 原始资料溯源
- [docs/raw/xxx.pdf]：原文页码/段落
- [docs/raw/xxx.md]：章节

## 7. 相关词条
[[关联概念1]] [[关联概念2]] [[衍生技术]]
```

**模板强制规则：**
- 所有 7 个板块 **必须完整**，缺一不可
- 「相关词条」板块 **不能为空**
- 每个板块有实质内容，不为凑模板留空
- 单文件不超过 3000 词，超长内容拆分子词条

### 3.3 双向内链

1. **内链格式**：`[[词条文件名不含后缀]]`
   - 例如 `[[LLM RAG]]` 而非 `[[LLM RAG.md]]`
2. **所有 wiki 页面** MUST 检查并补全内链
3. **新建词条** MUST 在所有关联的已有词条「相关词条」板块追加反向链接
4. **避免孤立页面**：每个词条至少被 2 个其他词条引用

### 3.4 raw 溯源规范

- 每条事实、结论、数据、引用**末尾加注** `[docs/raw/文件名.md:章节/行号]`
- 同一段落引用同一源，在段尾一次性标注，不逐句标注
- 多源混合段落：`[docs/raw/A.md:章节; docs/raw/B.md:章节]`

### 3.5 信息冲突标注

```markdown
> 【信息冲突】
> docs/raw/2026-karpathy-llm-wiki-原文.md 第 X 节提出：「……」
> 当前词条此前记载：「……」
> 两处描述存在矛盾，待进一步核实。
```

## Phase 4 — 全局一致性更新

### 4.1 更新 docs/wiki/index.md

1. **读取当前** `docs/wiki/index.md` 全文
2. 为新建词条**选择合适分类**，追加到对应分类下
3. 格式：
   ```markdown
   ## 分类名称
   - [[词条A]]
   - [[词条B]] ← 新增
   ```
4. 若无合适现有分类，**新建二级分类** `## 新分类名称`
5. 保持目录结构整洁，不出现空分类

### 4.2 追加 operation-log.md

在 `docs/wiki/operation-log.md` 末尾追加一条记录：

```markdown
YYYY-MM-DD | Ingest | 处理 docs/raw/{{文件名}}；新建词条：{{词条A}}、{{词条B}}；更新词条：{{词条C}}、{{词条D}}；【信息冲突】标记：{{冲突词条}}
```

格式严格保持：`日期 | 操作类型 | 内容明细`

**注意：**
- 日期取执行当日（`YYYY-MM-DD`）
- 操作类型固定为 `Ingest`
- 新建词条、更新词条用顿号分隔
- 无新建/无更新时对应位置写「无」
- 无冲突标记时省略该段

### 4.3 输出变更汇总

在完成所有操作后，以结构化清单输出所有变更：

````markdown
## Ingest 执行报告

### 处理文件
- docs/raw/{{文件名}}

### 新建词条（{{N}} 个）
每条格式：
- `docs/wiki/词条名.md`：一句话描述

### 更新词条（{{N}} 个）
每条格式：
- `docs/wiki/词条名.md`：补充了什么 / 修正了什么

### 【信息冲突】标记（{{N}} 处）
每条格式：
- `docs/wiki/词条名.md` 第X章：冲突概述

### 全局更新
- docs/wiki/index.md：{{分类调整说明}}
- docs/wiki/operation-log.md：已追加记录
````

## 强制规则（Violation = 不合格输出）

| 规则 | 说明 |
|------|------|
| **raw 只读** | 绝不修改、删除、重命名 `docs/raw/` 下任何文件，即使发现错误 |
| **无冗余词条** | 同一概念不创建第二个词条，必须更新已有词条 |
| **模板完整** | 新建词条 7 个板块缺一不可 |
| **冲突不取舍** | 遇到矛盾信息，标注并存，不擅自判断对错 |
| **溯源必标** | 每条结论绑定 docs/raw/ 文件位置 |
| **单文件 ≤3000 词** | 超长自动拆分子词条，通过 `[[父概念_子概念]]` 链接 |
| **操作必记日志** | 任何变更不对齐 operation-log → 视为未完成 |
| **无孤立页面** | 每个新建词条至少被 2 个页面内链引用 |

## 常见错误

### 1. 没读完现有 wiki 就开始提取
→ 漏更新、重复创建。Phase 1 前 MUST 遍历所有 wiki 页面。

### 2. 新建词条遗漏「对比关联」或「相关词条」
→ 板块必须完整，这是内联网的基础。

### 3. 内链格式错误
→ `[[概念名]]` 不是 `[[概念名.md]]` 不是 `[概念名]`。

### 4. 忽视冲突，擅自合并
→ 所有冲突必须标记，不自行判断。多篇文献矛盾正常，保留各方观点。

### 5. 忘记追加 operation-log
→ 任何变更操作必须带日志记录。这是全流程最后闭环。

### 6. 溯源标注过于笼统
→ `[docs/raw/xxx.md]` 不够，必须到章节级别 `[docs/raw/xxx.md:第X节]`。
