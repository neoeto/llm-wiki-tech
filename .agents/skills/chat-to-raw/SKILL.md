# Chat-to-Raw（会话知识归档）

**严格遵循根目录 `agent-spec.md` 全部规则。**

将当前 session 对话中产生的高价值知识结构化提取，生成原始素材文档保存到 `docs/raw/chats/`，然后自动调用 Ingest 流程将其摄入 `docs/wiki/`。

## 先决条件

**CRITICAL — 开始前 MUST 先读取以下文件：**

1. **`agent-spec.md`** — 核心原则、文件命名规范、Ingest 工作流规则
2. **`.agents/skills/ingest/SKILL.md`** — 后续调用的 Ingest 完整流程
3. **`docs/wiki/index.md`** — 了解现有 wiki 覆盖范围，避免重复
4. **`docs/wiki/operation-log.md`** — 了解最近已归档内容
5. **`docs/raw/chats/` 目录下已有文件**（如存在）— 了解已有聊天归档，避免重复

> 反例自检：用户说「保存」却不问范围 → 可能保存了用户不想保留的内容。**先问范围，再提取。**

## 核心流程

```
用户要求保存 session 知识
       │
       ▼
┌─────────────────────────────────────────────────┐
│ Step 1: 范围确认（与用户对话）                    │
│ - 明确要保存的主题/范围                           │
│ - 确认文件标题和标签                             │
│ - 确定是否立即执行 Ingest                       │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│ Step 2: 读取 session 内容 + 深度分级              │
│ - 用 session_read 获取相关消息                    │
│ - 按用户指定的范围过滤                            │
│ - 对每个主题标注 L1/L2/L3 深度层级                │
│ - L3 主题：完整保留 L1→L2→L3 递进链              │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│ Step 3: 生成 raw 文档                            │
│ - 按 raw 文件规范格式化                          │
│ - ≤2000 词：单文件；>2000 词：主体 + 附录拆分     │
│ - 保存到 docs/raw/chats/                        │
│ - 用户确认内容后继续                             │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│ Step 4: 调用 Ingest                              │
│ - 执行 Ingest 工作流处理新生成的 raw 文件          │
│ - 将知识编译为 wiki 词条                          │
└─────────────────────────────────────────────────┘
```

## Step 1 — 范围确认

**这是最关键的一步。在读取任何 session 内容之前，必须先与用户确认归档范围。**

### 确认话术模板

```
了解，我来将本次对话中的知识归档到 LLM Wiki。

先确认几个事项：

**① 归档范围**
当前对话涉及以下主题：
- {{主题A}}：{{一句话描述}}
- {{主题B}}：{{一句话描述}}
- {{主题C}}：{{一句话描述}}

你希望全部归档，还是只选择部分主题？

**② 文件命名建议**
建议标题：{{建议文件名}}（格式：YYYY-MM-DD-chat-主题.md）
你觉得合适吗，或者你有更好的命名？

**③ Ingest 执行**
归档完成后，是否立即执行 Ingest 将素材编译为 wiki 词条？
- 是 → 生成后自动摄入
- 否 → 仅保存 raw 文件，后续手动 Ingest
```

### 范围类型

| 用户意图 | 对应操作 |
|---------|---------|
| 「全部保存」/「整个对话」 | 提取整个 session 所有知识实体 |
| 「把关于 X 的记下来」 | 仅提取与 X 主题相关的消息 |
| 「记住这个方案/结论」 | 仅提取最后 N 轮的消息内容 |
| 「这几个点都有用」 | 用户指定多个主题/消息范围 |
| 「只保存代码片段」 | 仅提取代码和技术实现部分 |

### 追问原则

- **范围问题一次性问完**，不挤牙膏
- 用户表述模糊时，列出具体的主题列表让对方勾选
- 命名建议给出 1-2 个选项让用户选择，不填空
- 用户确认前 **不读取任何 session 内容**

## Step 2 — 读取 Session 内容

### 消息源获取

使用 `session_read` 或其他可用工具获取当前 session 消息历史：

```
可用工具（按优先级）：
1. session_read(session_id, include_todos=true) — 获取完整消息
2. session_search(query, session_id) — 搜索特定内容
```

### 范围过滤

根据 Step 1 确认的范围，从会话中提取对应消息：

| 范围类型 | 过滤策略 |
|---------|---------|
| 全量 | 所有 user ↔ assistant 消息 |
| 按主题 | 匹配主题关键词的消息轮次 |
| 按范围 | 指定起始消息 ID 或时间戳之后 |
| 按类型 | 仅代码块 / 仅结论 / 仅架构讨论 |

### 知识提取

从过滤后的消息中提取结构化信息：

| 提取类型 | 提取内容 |
|---------|---------|
| **技术决策** | 架构选型、方案对比、最终决策及理由 |
| **概念定义** | 新术语、名词解释、原理说明 |
| **示例与详解** | 代码示例、场景举例、对比示例、AI 先给方案后被用户纠正的过程 |
| **工作流/流程** | 操作步骤、命令序列、配置方法 |
| **问题&解决方案** | 遇到的问题、排查过程、最终修复 |
| **经验教训** | 踩坑记录、注意事项、最佳实践 |
| **数据/指标** | 量化结果、benchmark、配置参数 |
| **纠偏记录** | AI 给出方案后被用户纠正 → 完整记录错误版本、纠正过程、正确版本 |

### 对话深度分级（MUST 执行）

对每个涉猎主题，标注对话在该主题上挖到了哪个深度层级。**达到 L3 的主题，提取时必须保留 L1→L2→L3 的完整递进链，不可只留最终结论。**

| 深度 | 信号 | 必须保留的内容 |
|------|------|--------------|
| **L1 概念层** | 用户问「X 是什么」「介绍一下 X」 | 基础定义、一句话直觉理解 |
| **L2 原理层** | 用户追问「为什么」「怎么实现的」「详细讲讲」 | 架构/公式/机制层面的展开 |
| **L3 应用层** | 用户要求「给个例子」「代码怎么写」「对比一下」「和 Y 有什么区别」 | 具体示例、代码片段、对比表、纠偏过程 |

**分级示例**：
```
主题: LoRA 微调
├── L1: 用户问"LoRA是什么" → AI 给出直觉定义 ✓
├── L2: 用户追问"为什么低秩矩阵有效" → AI 展开数学原理 ✓
└── L3: 用户要求"给个代码示例，对比 QLoRA" → AI 给出代码 + 对比表 ✓

→ raw 文档必须包含 L1+L2+L3 三层完整内容，不可只记"LoRA 是低秩适配方法"。
```

### 内容质量标准

- **深度保留**：达到 L3 的主题，L1→L2→L3 完整递进链一个不丢
- **分级去冗余**：同一观点在**同一深度层级**反复出现 → 合并保留最完整版本；同一观点在**不同深度层级**出现 → 全部保留（概念本身 vs 原理展开 vs 示例应用是三个不同知识维度）
- **追问内容永不合并**：用户明确要求展开的内容（「详细讲讲」「给个例子」「那和 X 比呢」），其产生的示例、对比、代码、纠偏等内容视为独立知识点，不与其他内容合并
- **上下文完整**：确保每条知识独立可读，不依赖对话上下文。示例必须附带「这个示例说明什么」的一句话解释
- **保留不确定性**：如果某个结论当时存疑，在 raw 中标注「待验证」
- **标注贡献来源**：如知识来自用户提供，标注 `[来源：用户提供]`；如 AI 先给方案后被用户纠正，标注 `[经用户纠正]`

## Step 3 — 生成 raw 文档

### 文件命名规范

遵循 `agent-spec.md` §4 命名规则：

```
格式：年份-来源-标题.md
实际：YYYY-MM-DD-chat-<主题>.md
```

示例：
- `2026-07-03-chat-rag-architecture.md`
- `2026-07-03-chat-lora-vs-qlora.md`
- `2026-07-03-chat-weekly-summary.md`

### 目录结构

保存到 `docs/raw/chats/` 目录下：

```

└── chats/

    ├── 2026-07-03-chat-agent-workflow.md
```

若 `docs/raw/chats/` 不存在，自动创建。

### 内容长度控制

**主体 + 附录 拆分模式**：

| 文档类型 | 建议长度 | 内容 |
|---------|---------|------|
| **主体文档** | ≤2000 词 | 概述 + 技术决策 + 知识点（概念层）+ 附录索引 |
| **示例附录** | 不限 | L3 应用层展开的示例、代码、对比表、纠偏记录 |
| **QA 附录** | 不限 | 问答纠偏过程、排查路径、踩坑记录 |

**拆分规则**：
1. 主体文档达到 2000 词时，将 L3 层内容（示例、对比、代码）拆分到附录文件
2. 附录命名：`YYYY-MM-DD-chat-主题-appendix-类型.md`
   - 例：`2026-07-16-chat-lora-deep-appendix-examples.md`
   - 例：`2026-07-16-chat-lora-deep-appendix-qa.md`
3. 主体文档末尾标注附录索引：
   ```markdown
   ## 附录
   - 📎 示例与代码详见：[附录-示例](./2026-07-16-chat-lora-deep-appendix-examples.md)
   - 📎 纠偏与答疑详见：[附录-QA](./2026-07-16-chat-lora-deep-appendix-qa.md)
   ```
4. 若对话简短（总内容 ≤2000 词），不拆附录，单文件完成

**反例**：一刀切裁剪 → 示例或纠偏信息丢失。超过 2000 词时必须拆附录而非截断内容。

## 强制规则（Violation = 不合格归档）

| 规则 | 说明 |
|------|------|
| **先问范围** | 在读取任何 session 内容前，必须先与用户确认归档范围 |
| **不偷渡隐私** | 不保存用户明确排除的内容、个人信息、敏感信息 |
| **预览再写** | 写入前必须展示文档摘要给用户确认，不跳过此步骤 |
| **raw 只写不删** | 写入 `docs/raw/chats/` 后绝不修改、删除，即使用户说「撤回」|
| **命名合规** | 文件名必须遵循 `YYYY-MM-DD-chat-<主题>.md` 格式 |
| **不污染 docs/raw/** | 归档文件统一放在 `docs/raw/chats/` 子目录，不放在 `docs/raw/` 根目录 |
| **深度保留** | 每个主题标注 L1/L2/L3 深度层级；达到 L3 的主题，完整递进链一个不丢 |
| **分级去冗余** | 同深度层级内合并，不同深度层级间的展开全部保留。用户追问产生的内容永不合并 |
| **示例不丢** | 用户要求展开的示例、对比、纠偏过程作为独立知识点保留，不允许在「去冗余」时删除 |
| **拆分不截断** | 超过 2000 词必须拆主体+附录，不允许截断内容来满足长度限制 |
| **Ingest 可选** | 是否执行 Ingest 由用户决定，非强制 |

## 常见错误

### 1. 不问范围直接归档
→ 可能保存了用户不想保留的试错过程、敏感信息。**必须先问。**

### 2. 不预览直接写入
→ 用户无法控制内容。**预览是不可跳过的环节。**

### 3. 文件直接放 docs/raw/ 根目录
→ 污染 docs/raw/ 根目录。专门用 `docs/raw/chats/` 做隔离。

### 4. 保存后不告知用户后续操作
→ 用户不知道文件存在哪里、下一步做什么。**结束时必须说明文件位置和后续选项。**

### 5. 示例和追问内容被「去冗余」删除
→ 用户追问了三层（概念→原理→示例），归档时只留了概念定义，示例全丢了。**不同深度层级不合并，追问产生的示例/纠偏永不裁剪。**

### 6. 对话深度不标注
→ 归档后无法判断哪些主题经过了深入讨论。**每个主题 MUST 标注 L1/L2/L3 深度。**

### 7. 超过 2000 词直接截断
→ 后半段的示例和纠偏内容丢失。**必须拆主体+附录，不可截断。**

### 8. 代码/示例只有裸代码，缺上下文
→ 保存的示例必须附带「这个示例说明什么」的解释和消息来源标注。
