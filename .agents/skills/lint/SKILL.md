# Lint（知识库自检修复）

**严格遵循根目录 `agent-spec.md` 全部规则，尤其是 §3.3 Lint 知识库自检修复流程。**

对整个 `docs/wiki/` 知识库执行完整体检修复。建议每周执行一次，或在大量 Ingest 操作后立即执行。

## 先决条件

**CRITICAL — 开始修复前 MUST 先完整读取以下文件：**

1. **`agent-spec.md`** — 核心原则、标准词条模板、三大工作流规则
2. **`docs/wiki/index.md`** — 全局目录，检查分类和条目完整性
3. **`docs/wiki/operation-log.md`** — 了解历史变更，判断上次 Lint 时间
4. **`docs/wiki/` 下所有 `.md` 文件** — 遍历全部词条页面，逐一扫描

> 反例自检：只扫了部分文件就开始修复 → 遗漏孤立页面、漏掉冲突标记。**全量遍历在先，逐项修复在后。**

## 核心流程

```
全量遍历 docs/wiki/ 所有 .md 文件（不含 index.md / operation-log.md）
      │
      ▼
┌─────────────────────────────────────────────┐
│ Phase 1: 扫描诊断（只读，不动文件）           │
│ ① 孤立页面检测                               │
│ ② 内容重复检测                               │
│ ③ 超长页面检测                               │
│ ④ 缺失板块检测（溯源 / 相关词条）             │
│ ⑤ 信息冲突标记完整性检查                      │
│ ⑥ 失效内链检测                               │
│ ⑦ 分类目录健康检查                            │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│ Phase 2: 修复执行                             │
│ - 按优先级逐项修复                            │
│ - 每修复一项立刻验证                          │
│ - 不可修复的生成诊断报告                      │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│ Phase 3: 全局刷新 + 报告                      │
│ - 更新 docs/wiki/index.md                    │
│ - 追加 docs/wiki/operation-log.md            │
│ - 输出完整修复报告                            │
└─────────────────────────────────────────────┘
```

## Phase 1 — 全量扫描诊断（只读）

### ① 孤立页面检测

**定义**：一个 wiki 词条页面未被任何其他 wiki 页面通过 `[[词条名]]` 引用。

**检测方法**：
1. 收集所有词条文件名 → 提取 `basename .md` 作为概念名
2. 对所有 `.md` 文件（不含 `index.md`、`operation-log.md`）运行正则：`\[\[概念名\]\]`
3. 统计每个概念名被引用的次数
4. 引用次数 = 0 的标记为 **孤立页面**

**扫描记录**：

| 孤立词条 | 文件路径 | 被引用次数 | 应链接到的相关词条 |
|---------|---------|-----------|------------------|
| {{概念X}} | docs/wiki/xxx.md | 0 | [[A]]、[[B]] |

### ② 内容重复检测

**检测方法**：
1. 提取每个词条的「基础定义」段落
2. 对任意两个词条的「基础定义」做语义/关键词重叠比对
3. 对「核心原理」章节对比，识别超过 40% 篇幅实质相同的段落
4. 检查是否存在 **同一概念多个词条**（如 `RAG.md` 和 `LLM RAG.md` 并存）

**检测结果分类**：

| 严重程度 | 定义 | 处理方式 |
|---------|------|---------|
| **完全重复** | 同一概念独立成两个词条 | 合并至主词条，删除冗余 |
| **实质重叠** | 两词条 50%+ 内容重复但侧重不同 | 去重后保留差异部分，主词条汇总 |
| **概念别名** | 同一概念不同命名（如 LoRA / Low-Rank Adaptation） | 重定向或别名词条内链引用 |

### ③ 超长页面检测

**检测方法**：
1. 使用 `wc -w` 统计每个 `.md` 文件的词数
2. 标记 **≥3000 词** 的页面

**拆分规则**：
- 按「核心原理」中的子主题自然切割
- 子词条命名格式：`主概念_子主题.md`（如 `Transformer_Attention.md`、`Transformer_PositionEncoding.md`）
- 主词条保留概要 + 指向子词条的 `详细内容见 [[主概念_子主题]]`
- 子词条严格套用标准模板，并在「相关词条」引用主词条

### ④ 缺失板块检测

**扫描每个词条的板块完整性**，检查以下缺失情况：

| 板块 | 检测规则 | 检查方法 |
|------|---------|---------|
| **原始资料溯源**（§6） | 是否存在 `## 6.` 标题 | grep `## 6.` |
| 溯源内容非空 | `grep '\[docs/raw/'` 该章节是否有条目 |
| **相关词条**（§7） | 是否存在 `## 7.` 标题 | grep `## 7.` |
| 相关词条非空 | 标题下是否有 `[[...]]` |
| **对比关联**（§5） | 是否存在 `## 5.` 标题 | grep `## 5.` |

### ⑤ 信息冲突标记完整性检查

**扫描所有 `【信息冲突】` 标记**，对每个标记执行：

1. 确认冲突双方是否都有 `[docs/raw/...]` 溯源
2. 确认标注了冲突的 docs/raw 文件路径和章节
3. 若某一方的溯源已丢失 → 标记为 **「悬挂冲突」**，需补充
4. 若冲突已可通过新 raw 素材解决 → 更新标记或移除

**扫描记录**：

| 词条 | 冲突位置 | 双方是否溯源 | 状态 |
|------|---------|------------|------|
| [[词条A]] | §2.3 | docs/raw/X.md:章 / docs/raw/Y.md:章 | ✅ 完整 |
| [[词条B]] | §1 | docs/raw/Z.md:章 / 无溯源 | ❌ 悬挂 |

### ⑥ 失效内链检测

**检测方法**：
1. 对所有 `.md` 文件提取全部 `[[概念名]]` 引用
2. 检查 `docs/wiki/概念名.md` 是否存在
3. 不存在的标记为 **失效内链**

**扫描记录**：

| 来源词条 | 失效内链 | 建议操作 |
|---------|---------|---------|
| docs/wiki/A.md | [[不存在概念]] | 删除/替换/新建该词条 |

### ⑦ 分类目录健康检查

**检测方法**：
1. 取 `docs/wiki/index.md` 中所有 `- [[词条名]]` 条目
2. 与 `docs/wiki/` 下实际 `.md` 文件列表对比（排除 index.md、operation-log.md）
3. **漏列**：实际存在但 index.md 未收录 → 追加
4. **幻影**：index.md 收录但文件不存在 → 删除/修正
5. **错分类**：词条分类明显不合适 → 调整

## Phase 2 — 修复执行

### 修复优先级

| 优先级 | 修复类型 | 说明 |
|--------|---------|------|
| P0 | **失效内链** | 导致导航断裂，必须先修 |
| P0 | **完全重复** | 内容冗余严重，立即合并 |
| P1 | **缺失板块**（溯源/相关词条） | 质量缺陷，尽快补全 |
| P1 | **超长页面** | 影响可维护性，拆分子词条 |
| P1 | **孤立页面** | 知识孤立，补充内链 |
| P2 | **悬挂冲突** | 信息缺失，补充或标记 |
| P2 | **分类目录异常** | 索引不准确，刷新即可 |
| P3 | **实质重叠** | 优化性质，可安排在低负载时 |

### 2.1 孤立页面修复

1. 阅读孤立页面全文，提取其核心概念和关联实体
2. 在**所有相关词条**的「相关词条」板块追加 `[[孤立概念]]`
3. 确保该概念被至少 2 个其他页面引用

```markdown
## 7. 相关词条
（追加）[[孤立概念A]]
```

### 2.2 重复词条合并

**完全重复合并步骤**：

1. 确定**主词条**（通常保留创建更早、内容更完整、命名更规范的）
2. 将**待删除词条**中主词条未覆盖的内容迁移至主词条对应板块
3. 在 `docs/wiki/operation-log.md` 记录合并
4. 文件中做注释标记合并去向

```
⚠️ 该词条内容已合并至 [[主词条名]]，本页面保留引用跳转指向。
```

**实质重叠处理步骤**：

1. 识别重叠段落，在主词条中合并保留
2. 差异部分保留在各自词条中，互加内链
3. 在「对比关联」板块注明差异定位

### 2.3 超长页面拆分

**标准步骤**：

1. 识别自然拆分边界（按一级标题 `##` 章节）
2. 每个子主题创建 `主概念_子主题.md`
3. 子词条套用标准模板：
   - 基础定义：精确定义该子主题
   - 核心原理：原章节完整内容 + 展开
   - 原始资料溯源：继承主词条对应部分的溯源
   - 相关词条：必须包含 `[[主概念]]` + 其他关联概念
4. 主词条对应章节改为概要 + 跳转链接：

```markdown
## 2. 核心原理

### 2.X {{子主题名}}
（概要 1-2 段）
详细内容见 [[主概念_子主题]]
```

### 2.4 缺失板块补全

**缺失溯源**：
- 若该词条最初来源于某个 raw 文件：追加 `[docs/raw/文件名.md:概述]`
- 若无法追溯来源：标注 `[来源待补充]`

**缺失相关词条**：
- 阅读全文，提取所有直接关联概念
- 关联标准：共同主题、技术依赖、对比关系、同作者
- 至少写 3-5 个，避免单链

**缺失对比关联**：
- 识别该概念所属技术类别
- 寻找同级同类概念做对比
- 至少写 1-2 组对比关系

### 2.5 信息冲突修复

对 Phase 1 ⑤ 检出的每个冲突：

| 状态 | 处理 |
|------|------|
| ✅ **双方溯源完整** | 保留原样，不做取舍 |
| ❌ **悬挂冲突**（一方无溯源） | 补充说明「当前该方观点缺少 raw 溯源，待核实」 |
| ✅ **已可通过新素材解决** | 更新冲突标记，追加新素材观点，标注时间 |
| ⚠️ **标记格式不规范** | 统一为 Phase 3 的规范格式 |

### 2.6 失效内链修复

对 Phase 1 ⑥ 检出的每条失效内链：

| 情况 | 处理 |
|------|------|
| 链接概念有同义词/别名词条 | 替换为 `[[正确词条名]]` |
| 链接概念确实不存在且不应存在 | 删除该内链 |
| 链接概念确实不存在但应有词条 | 标记为**待新建**，在报告中列出，询问用户是否补充 raw 素材 |

### 2.7 分类目录刷新

对 Phase 1 ⑦ 检出的每个问题：

- **漏列**：追加 `- [[词条名]]`
- **幻影**：删除该行
- **错分类**：移动至正确分类下
- **空分类**：移除（或标注「待补充」）

## Phase 3 — 全局输出

### 3.1 追加 docs/wiki/operation-log.md

```markdown
YYYY-MM-DD | Lint | 体检修复：孤立页面 {{N}} 个→已处理；重复词条 {{N}} 组→{{合并/待处理}}；超长页面 {{N}} 个→拆分子词条；补充缺失溯源 {{N}} 处、缺失相关词条 {{N}} 处；【信息冲突】处理 {{N}} 处；失效内链 {{N}} 条→已修复；刷新 index.md 目录。
```

若无某项修复，对应位置写「无」。

### 3.2 完整修复报告

````markdown
## Lint 修复报告

执行日期：YYYY-MM-DD

### ① 孤立页面
- 检出：{{N}} 个
- 已处理：{{N}} 个
- 处理明细：
  - `docs/wiki/页面A.md` → 追加内链至 [[页面B]]、[[页面C]]

### ② 重复词条
- 检出：{{N}} 组
- {{完全合并 N 组 / 实质重叠 N 组}}
- 处理明细：
  - [[主词条A]] ← [[冗余词条B]]：内容已合并，{{冗余词条B}} 保留跳转引用

### ③ 超长页面拆分
- 检出：{{N}} 个（{{页面名}}：{{实际词数}}词）
- 已拆分：{{N}} 个 → 生成 {{N}} 个子词条
- 处理明细：
  - `docs/wiki/主概念.md` → 拆出 `docs/wiki/主概念_子主题1.md`、`docs/wiki/主概念_子主题2.md`

### ④ 缺失板块补全
- 补充原始资料溯源：{{N}} 处
  - `docs/wiki/页面A.md` → 追加 [docs/raw/xxx.md]
- 补充相关词条：{{N}} 处
  - `docs/wiki/页面B.md` → 追加 [[C]]、[[D]]
- 补充对比关联：{{N}} 处

### ⑤ 信息冲突处理
- 冲突标记合计：{{N}} 处
- ✅ 双方溯源完整且格式规范：{{N}}
- ❌ 悬挂冲突（待核实）：{{N}}
- 处理明细：
  - `docs/wiki/页面X.md` §2.3 → 冲突双方均溯源完整，保留标注

### ⑥ 失效内链
- 检出：{{N}} 条
- 已修复：{{N}} 条
- 待建词条：{{N}} 条（{{概念名}}）
  - 需要补充 raw 素材后通过 Ingest 创建

### ⑦ 分类目录更新
- `docs/wiki/index.md`：漏列 +{{N}} / 幻影 -{{N}} / 错分类 ↻{{N}} / 空分类移除 {{N}}
````

## 强制规则（Violation = 不合格修复）

| 规则 | 说明 |
|------|------|
| **只读不修** | Phase 1 全程只读，不修改任何文件。修复在 Phase 2 统一执行。 |
| **先扫全量** | 不能边扫边修，必须全量诊断完再执行修复，防止诊断被修复动作干扰。 |
| **不擅自删除** | 合并词条时原文件保留引用跳转，不彻底删除。除非用户明确确认。 |
| **冲突不篡改** | 信息冲突标记只做完整性检查，不篡改冲突双方观点。 |
| **日志必记** | 任何修复动作未记录到 docs/wiki/operation-log → 视为未完成。 |
| **子词条模板完整** | 拆分子词条也必须套用标准模板 7 板块。 |
| **孤立页面 ≥2 引用** | 修复后每个原孤立词条必须被至少 2 个页面引用。 |
| **不预判 raw 内容** | 不能根据 raw 文件名猜测内容来补全溯源。 |

## 常见错误

### 1. Phase 1 未遍历全量就开始修
→ 诊断不完整导致遗漏。Phase 1 遍历全部 `.md` 文件是最低要求。

### 2. 合并时直接删除文件
→ **绝不直接删除**，保留引用跳转页，用户确认后再清理。

### 3. 超长页面强制一刀切
→ 3000 词是上限，不是目标。如果内容紧凑不可拆分，保留并备注。按自然边界拆分，不机械断章。

### 4. 孤立页面只加一个内链
→ 规则要求 ≥2 引用，只加一个等于半修。

### 5. 拆分子词条后忘记在 docs/wiki/operation-log 记录
→ 拆分是新文件创建，必须记录。

### 6. 修复时不区分 P0-P3 优先级
→ 失效内链和完全重排必须优先于格式优化类修复。

## 与 Ingest、Query 的联动

```
Ingest 批量导入 → docs/wiki 词条增多
       │
       ▼
  定期执行 Lint → 发现孤立/重复/超长/缺失
       │
       ▼
  修复完成后 → 串联完整、可高效 Query
       │
       ▼
  Query 时发现知识缺口 → 触发新 Ingest
```

三个 skill 形成闭环：**Ingest 建 → Lint 修 → Query 用 → 缺口反馈 → Ingest 补**。
